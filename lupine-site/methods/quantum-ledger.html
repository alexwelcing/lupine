The Reproducibility Problem

The publication of machine-learning molecular dynamics results faces a structural reproducibility crisis. Unlike classical force-field simulations with deterministic initial conditions, NNFF-driven trajectories carry an additional layer of non-reproducibility: model version, hardware floating-point behavior, batch parallelization order, and PRNG seed all affect the precise trajectory — even when the input structure and potential parameters appear identical.

Lupine's Quantum Ledger is our answer. It is a cryptographically-linked provenance layer embedded inside the glimPSE simulation engine that records every inference step of every simulation run — from force predictions and energy evaluations to the hardware context and model checkpoint hash used at runtime.

Ledger Entry Schema

Each simulation step writes a single Ledger Entry to the in-memory append-only log. At trajectory export, entries are serialized and embedded in the XDATCAR file header as a Base64-encoded JSON block.

LEDGER ENTRY SCHEMA // STEP 1,247 
{
  "step_id": 1247,
  "timestamp_ns": 1710432847293810432,
  "potential": "CHGNet@v0.3.3.sha256:a4f9c2",
  "energy_eV": -847.234192,
  "max_force_eV_A": 0.04128,
  "temperature_K": 2400.0,
  "hardware_ctx": "WebGPU/Apple M3 Pro",
  "prev_hash": "sha256:7b2c4f…",
  "entry_hash": "sha256:9e3f1a…"
}

Each entry is chained to the previous via its SHA-256 hash, forming a tamper-evident linked list. Any modification to a historical step will invalidate every subsequent hash — making post-hoc trajectory manipulation detectable without a trusted third party.

Quantum Ledger Synchronization

When a simulation completes in glimPSE, the Ledger Manifest is written to Lupine's distributed Ledger network — a content-addressed storage layer built on IPFS. This gives every Lupine simulation a permanent, globally-addressable proof of computation that can be cited in publications:

Lupine Ledger URI // Permanent Reference
lupine://ledger/sha256:9e3f1a7c4d8b2e6f…

Reviewers and collaborators can independently verify a simulation's integrity by fetching the manifest and re-running the trajectory from its recorded initial conditions through the pinned WASM bundle version referenced in the manifest header.
